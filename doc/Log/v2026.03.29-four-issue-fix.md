# v2026.03.29 - Four-Issue Fix (Image, Agent Chain, Memory, Kimi)

## 概述

本次更新修复了 4 个问题/需求：图片识别慢、Agent 思维链不显示、跨会话 Memory 功能、Kimi 模型 thinking+tool_call 冲突。

## 改动内容

### 1. 图片识别性能优化

**问题：** 上传图片后 AI 识别非常慢，因为图片以原始大小的 base64 发送（最大 10MB），无压缩无缩放。

**实现：**
- 新增 `client/services/imageUtils.ts`：基于 Canvas 的客户端图片压缩工具，最大边长限制 1568px，JPEG 质量 0.85，PNG 保持格式。
- `client/App.tsx` 的 `processFile` 中，图片在存入 attachment 前先经过 `compressImage()` 压缩。
- 所有 `image_url` 对象（`backendService.ts`、`externalApiService.ts`、后端 `chat.ts`）新增 `detail: 'high'` 参数，告诉 API 使用高分辨率模式但图片本身已合理缩放。

### 2. Agent 模式思维链实时显示

**问题：** Agent loop 中只输出最终回复，中间的 tool call/result 过程在 loop 结束后才一次性插入 UI。

**实现：**
- `agentRuntime.ts` 新增 `onToolMessages` 回调（`AgentRunInput` 接口）。
- 每个 iteration 结束后立即调用 `onToolMessages` 将 toolCallMsg 和 resultMsg 发送给 UI。
- `App.tsx` 实现 `onToolMessages` 回调，将中间消息实时插入到 bot 占位消息前面，用户可以看到每一轮 tool 调用和结果。
- `buildToolCallMessage` 新增 `reasoning` 参数，保存中间 iteration 的思维内容，由 `ThinkingBlock` 渲染。

### 3. Memory 功能（跨会话、跨设备）

**问题/需求：** Agent 缺少持久化记忆，无法跨 session 记住用户偏好、任务等上下文。

**后端：**
- 新增 PostgreSQL 表 `ai_studio_memory`（id, user_id, content, tags JSONB, category, source_session_id, timestamps）。
- 新增路由 `src/routes/aiStudioMemory.ts`：GET/POST/PUT/DELETE + bulk-import，全部需认证，按 user_id 隔离。
- 迁移脚本加入 `src/db/migrate-postgres.ts`。

**前端：**
- 新增 `client/services/memoryService.ts`：API 客户端 + `exportToMarkdown` / `parseMarkdown` / `downloadMarkdown` / `importFromFile`。
- 新增 `client/skills/memory.skill.md`：定义 4 个工具（memory_read, memory_write, memory_update, memory_delete）+ Agent 使用指南。
- `builtinTools.ts` 注册 4 个 memory 工具执行器。
- `skillsRegistry.ts` 注册 Memory skill 为内置 skill。
- `AgentConfigPanel.tsx` 新增 Memory 面板（在 Agent 设置里），支持：查看 memory 列表、删除条目、导入 memory.md、导出 memory.md。

**memory.md 格式：** 自然散文式 Markdown，每条 `---` 分隔，包含 category、日期、内容、tags。

### 4. Kimi 模型 Thinking + Tool Call 冲突修复

**问题：** Kimi (Moonshot) 开启 thinking 时调用工具会报 `thinking is enabled but reasoning_content is missing in assistant tool call message`。

**原因：** `toOpenAIHistory` 构建的 assistant 消息（含 tool_calls）没有 `reasoning_content` 字段，Kimi 要求 thinking 模式下所有 assistant 消息都必须有此字段。

**实现：**
- `toOpenAIHistory` 新增 `enableThinking` 参数，当为 true 时在所有 assistant 消息上添加 `reasoning_content`（有则填值，无则空字符串）。
- Agent loop 中两处 `toOpenAIHistory` 调用均传入 `input.enableThinking`。
- `buildToolCallMessage` 保存中间 reasoning 到 Message 上，确保后续构建历史时有 reasoning 可用。
