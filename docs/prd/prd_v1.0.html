一、需求背景

1.1 现状问题

问题一：AI 工具碎片化严重。用户需要在 ChatGPT、Claude、Gemini 等多个平台间反复切换，每个平台有独立的账号体系和对话历史，无法统一管理。非技术用户面临认知负担大、学习成本高的困境。

问题二：AI 能力止步于"对话"。主流 AI 工具以纯文本对话为主，缺乏可扩展的"行动能力"。用户无法让 AI 主动调用外部工具（如查日历、发邮件、搜索文件），只能手动复制粘贴信息。

问题三：应用间数据孤岛。用户在不同应用（日历、桌面、笔记）中产生的数据彼此隔离，AI 无法获得完整的用户上下文。例如日历中有明天的会议，但聊天 AI 无法主动提醒用户准备材料。

问题四：现有方案不支持自部署和自定义。企业和个人用户对数据隐私有顾虑，希望使用自有 API Key、部署在自己的服务器上，但现有 SaaS 产品无法满足这一需求。

1.2 为什么现在做？

1) 技术窗口期成熟：多家 AI 厂商已提供稳定的 API（Gemini 2.x、GPT-4o、Claude 3.5、DeepSeek V3），SKILL.md 微型 Agent 架构已在 v0.4.4 中验证可行。

2) Forsion 生态基础已建立：后端统一服务（forsion-backend-service）已完成用户认证、积分计费、多项目模型配置等基础能力，具备跨应用数据整合的技术条件。

3) 用户需求明确：v0.4.x 实验版本的使用反馈表明，多模型切换和个性化主题是用户高频使用的功能，Skills 扩展能力是最期待的新方向。

二、需求目标

目标类型描述衡量指标目标值
核心业务目标 提供统一的多模型 AI 对话入口，让用户无需在多个平台间切换 用户首选使用率（将 AI Studio 作为主要 AI 工具的用户比例） ≥ 60%
差异化目标 通过 Skills 体系赋予 AI "行动能力"，超越纯对话 Skills 安装量 & 使用频次（每用户平均启用 Skills 数） ≥ 3 个/用户
生态目标 通过跨应用数据中枢打通 Forsion 生态，形成用户粘性 跨应用 Skill 调用率（涉及其他 App 数据的对话比例） ≥ 20%
用户体验目标 非技术用户 5 分钟内完成首次对话 + Skill 启用 新用户 onboarding 完成率（从注册到首次成功使用 Skill） ≥ 80%
技术目标 支持私有化部署，用户可使用自有 API Key Docker 一键部署成功率 ≥ 95%

三、用户与使用场景

3.1 典型用户画像

小美 — 内容创作者
  • 28 岁，自媒体运营
  • 每天需要写文章、翻译、做社交媒体文案
  • 不懂技术，希望 AI 工具"开箱即用"
  • 核心诉求：一个地方搞定所有 AI 需求，不想来回切换平台
阿杰 — 小型团队负责人
  • 35 岁，创业公司 CEO
  • 需要团队统一使用 AI，控制成本
  • 关心数据安全，希望自部署
  • 核心诉求：统一管理团队 AI 使用，积分计费透明可控
老陈 — Forsion 生态深度用户
  • 42 岁，项目经理
  • 同时使用 Forsion Desktop 和 Calendar
  • 希望 AI 能理解自己的工作全貌
  • 核心诉求：AI 能自动关联日历日程，在聊天中提醒和辅助决策

3.2 核心用户旅程图

阶段 用户触点 用户行为 痛点 / 情绪 机会点 / 功能转化
1. 发现 官网 / 推荐链接 了解产品，决定试用 不确定是否又是一个普通 AI 工具 首页突出"多模型 + Skills + 跨应用"差异化价值
2. 注册 登录/注册页 使用邀请码注册账号 注册流程繁琐则直接放弃 极简注册（用户名 + 密码 + 邀请码），获赠初始积分
3. 首次对话 主聊天界面 选择模型，发送第一条消息 不知道选哪个模型好，怕选错浪费积分 默认推荐最佳模型，首次对话 0 积分消耗（引导体验）
4. 探索 Skills Settings > Skills 市场 浏览并安装感兴趣的 Skills 不理解 Skill 是什么，概念模糊 每个 Skill 卡片配有一句话描述 + 使用演示，安装即用
5. 深度使用 聊天中调用 Skill 让 AI 联网搜索 / 读取日历 / 执行代码 Skill 失败时不知道原因 Skill 执行过程可视化，失败时给出清晰原因和重试引导
6. 跨应用联动 对话中涉及其他 App 数据 问 AI "明天有什么安排" → AI 读取 Calendar 数据 数据授权流程需要清晰，用户担心隐私 首次跨应用访问弹出授权确认，一次授权永久有效（可撤销）
7. 持续留存 日常工作流 将 AI Studio 作为日常工具，管理多个会话 对话历史管理混乱，找不到之前的重要对话 会话分类 + Emoji 标签 + 搜索 + 置顶功能

交互流程

flowchart TD
  openSettings["用户打开 Settings"] --> clickTab["点击 Skills Tab"]
  clickTab --> showMarket["展示 Skills 市场"]
  showMarket --> action{用户操作}
  action --> search["搜索关键词"]
  action --> filterBtn["筛选: 全部/官方/已安装"]
  action --> viewCard["查看 Skill 卡片"]
  action --> createBtn["创建自定义 Skill"]
  search --> filtered["实时过滤列表"]
  filterBtn --> filtered
  filtered --> viewCard
  viewCard --> isInstalled{已安装?}
  isInstalled -- 否 --> install["点击安装"]
  isInstalled -- 是 --> scope["管理作用域"]
  install --> isCross{跨应用?}
  isCross -- 否 --> done["安装成功 · 全局启用"]
  isCross -- 是 --> authTag["安装成功 · 标记需授权"]
  done --> scope
  authTag --> scope
  createBtn --> editor["打开 SkillEditor"]
  editor --> saveDone["保存 → 安装"]

业务规则

  • Skills 市场作为 SettingsModal 的第 4 个 Tab（skills），位于 general 和 account 之间
  • 卡片展示字段：图标、名称、一句话描述、工具数量、官方 / 跨应用 / 已安装徽章
  • 搜索支持名称和描述的模糊匹配，实时过滤（debounce 300ms）
  • 三个筛选条件互斥：全部（默认）/ 仅官方 / 已安装
  • 安装操作将 Skill 加入用户已激活列表，默认作用域为全局
  • 卸载操作将 Skill 从已激活列表移除，IndexedDB 中的自定义 Skill 数据保留 30 天
  • 跨应用 Skill 安装后显示“需授权”徽章，实际授权延迟到首次调用时触发
  • 每个用户最多创建 20 个自定义 Skill
  • 官方 Skill 列表优先从后端 GET /api/skills/catalog 获取；离线时降级为本地 skillsRegistry

P0 5.2 Skill 作用域控制

交互流程

flowchart TD
  installed["已安装 Skill"] --> scopeUI["作用域管理面板"]
  scopeUI --> choice{选择作用域}
  choice --> global["全局启用"]
  choice --> session["仅当前 Session"]
  choice --> disabled["已禁用"]
  global --> save["保存到 localStorage"]
  session --> save
  disabled --> save
  save --> effect["立即生效于后续消息"]
  global -.- noteG["所有新 Session 自动注入该 Skill"]
  session -.- noteS["仅在当前 Session Agent 配置中启用"]
  disabled -.- noteD["所有 Session 均不可用"]

业务规则

  • 三个作用域级别：全局（所有 Session 默认启用）、仅当前 Session、已禁用
  • 安装后默认作用域：全局
  • 作用域变更仅影响后续新消息，不回溯修改已有消息
  • 全局作用域：Skill 的 System Prompt 自动注入每个新 Session
  • Session 作用域：仅当用户在该 Session 的 AgentConfigPanel 中手动启用时注入
  • Workspace Skill（会话上下文管理）始终自动注入，不允许设置作用域
  • 作用域设置存储在 localStorage（以 Skill ID 为 key），后续云同步版本将迁移到后端

P1 5.3 Skill 执行可视化

交互流程

flowchart TD
  msg["用户发送消息"] --> ai["AI 分析意图"]
  ai --> needTool{需要调用 Tool?}
  needTool -- 否 --> textReply["直接生成文本回复"]
  needTool -- 是 --> execState["显示执行中状态：Spinner + 工具名"]
  execState --> result{执行结果}
  result -- 成功 --> success["绿色对勾 + 工具名 + 耗时"]
  result -- 失败 --> fail["红色叉 + 工具名 + 错误原因 + 重试按钮"]
  result -- 超时 --> timeout["超时提示 + 重试/切换模型"]
  success --> expand["可点击展开详细返回数据"]
  expand --> combine["AI 结合结果生成回复"]
  fail --> retry{用户点击重试?}
  retry -- 是 --> execState
  retry -- 否 --> textReply
  timeout --> retry

业务规则

  • Tool Call 状态展示位于消息流中（inline），不使用独立面板
  • 三种视觉状态：执行中（spinner + 工具名）、成功（绿色对勾 + 工具名 + 耗时）、失败（红色叉 + 工具名 + 错误信息 + 重试按钮）
  • 成功状态默认折叠，点击可展开查看完整 Tool 返回的 JSON 数据
  • 失败状态醒目展示错误原因，提供 [重试] 按钮
  • 超时阈值 30 秒，超时后显示超时提示 + [重试] + [切换模型]
  • 单条消息中多个 Tool Call 按执行顺序依次展示
  • 基于现有 ToolCallBlock 组件增强，保持与现有代码架构一致

    FE->>U: 弹出授权确认弹窗
    alt 允许
      U->>FE: 点击允许访问
      FE->>BE: 保存授权记录
    else 拒绝
      U->>FE: 点击拒绝
      FE->>AI: 授权被拒
      AI->>U: 提示需授权 + Settings 链接
    end
  end

  FE->>BE: 请求数据（user_id + auth_token）
  BE->>BE: 验证授权 + 查询用户数据分区
  BE->>APP: 读取对应 App 数据
  APP-->>BE: 返回数据
  BE-->>FE: 返回结构化数据
  FE->>AI: Tool 执行结果
  AI->>U: 基于数据生成回复

业务规则 — 用户级数据分区

  • 后端在用户首次登录时创建数据分区（表 user_data_partitions）
  • 每个分区按 App ID 划分命名空间（calendar、desktop、notes 等）
  • 数据结构：{ user_id, app_id, data_type, payload JSON, created_at, updated_at }
  • 读写权限由授权记录控制，未授权的 App 无法读取其他 App 的数据
  • 用户删除时级联清除所有分区数据

业务规则 — 跨应用 Skill 调用

  • 跨应用 Skill 是标准 SKILL.md 文件，Tools 使用 http executor 调用后端数据中枢 API
  • Skill frontmatter 须声明所需 App 权限：required_apps: [calendar]
  • v1.0 首批跨应用 Skill：Forsion Calendar Skill（get_events、create_event、update_event）
  • 后端 API 模式：GET/POST /api/data-hub/{app_id}/{resource}
  • Skill 调用时前端自动注入用户 auth_token，后端校验身份和授权

业务规则 — 数据授权管理

  • 授权弹窗仅在用户首次访问某 App 数据时触发（每 App 每用户一次）
  • 弹窗内容：App 名称 + App 图标 + 权限明细列表 + 隐私说明
  • 用户选择：“允许访问”（永久有效直到撤销）或 “拒绝”
  • 拒绝后：AI 收到拒绝信号，回复提示用户可在 Settings 中授权
  • 撤销方式：Settings > 数据授权管理（新增 Tab 或子页面），列出已授权 App，可逐个撤销
  • 撤销后下次访问将重新弹出授权弹窗
  • 授权记录存储在后端：{ user_id, app_id, granted_scopes[], granted_at }

交互流程

flowchart TD
  appLoad["应用加载"] --> checkLocal{本地有 localStorage 会话?}
  checkLocal -- 是 --> checkBackend{后端可达?}

  autoSave --> syncOK{保存成功?}

业务规则

  • 迁移策略：首次连接后端时，将现有 localStorage 中的会话批量上传；成功后标记迁移完成（不删除本地，仅标记）
  • 后端 API：会话和消息的 CRUD 端点（/api/chat/sessions、/api/chat/messages）
  • 冲突解决：Last-Write-Wins（以服务端时间戳为准）
  • 自动保存：每条消息发送/接收后 2 秒内触发后端保存（debounce 合并）
  • 离线支持：网络不可用时继续使用 localStorage，变更加入队列，重连后批量同步
  • 同步数据范围：消息内容、会话标题、emoji 标签、Agent 配置、归档状态
  • 会话上限：保留最近 100 个活跃会话，超出自动归档最早的
  • 消息分页：每个会话默认加载最近 50 条消息，上滑加载更多

业务规则

  • 鉴权：使用用户的 auth_token（从前端传递），后端校验身份
  • 每个 App 自定义数据 Schema（Calendar: events，Desktop: file_index 等）
  • 两种同步模式：full（替换全部数据）、incremental（新增/更新指定记录）
  • 频率限制：每用户每 App 每分钟最多 100 次同步请求
  • 后端按已注册的 App Schema 校验 payload 格式
